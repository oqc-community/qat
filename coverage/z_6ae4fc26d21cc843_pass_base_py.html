102class PassManager(PassInfoMixin): 

103 """Contains a sequence of passes. 

104 

105 Represents a sequential composite of passes, which can be composites themselves. 

106 Although not explicitly specified, the pass manager is best modelled as a DAG that acts 

107 as a pass itself. In doing so, it runs a sequence of (composite or leaf) passes over the 

108 input and aggregates results from them. 

109 

110 Result aggregation (in general) can be achieved via return values or side effects. We 

111 adopted the latter approach where a :class:`ResultManager` is passed in as argument. The 

112 rationale behind this choice is two-fold: 

113 

114 #. Allow passes to peek, use, or potentially invalidate results via the ResultManager 

115 without burdening their parent :class:`PassManager`. 

116 #. Let the passes and results define their dependency graphs on the fly which saves us 

117 from explicitly building and maintaining them. Here it's also a possible direction to 

118 let passes lazily declare, create, and run their dependencies as they get discovered. 

119 

120 Result aggregation would also potentially involve cache handling as described in the 

121 :class:`ResultManager`. This calls for proper pass identification, cache invalidation, 

122 and pass/result cycle detection techniques. 

123 

124 Example of a DAG: 

125 

126 (PM1 A B (PM2 C D ) E F) can be thought of as: 

127 

128 .. code-block:: 

129 

130 <---PM2---> 

131 

132 --> C 

133 A --> B --| ↓ 

134 --> D --> E --> F 

135 

136 <-----------PM1------------> 

137 

138 Although the notation of a given (sub-) DAG looks sequential, passes within can run 

139 sequentially or in parallel depending on their inter-dependencies. 

140 

141 Notes thus far describe an ideal fully-fledged PassManager. However, today's needs are 

142 very simple as we barely have visible features of modular quantum compilation workflow. 

143 It is therefore wise to keep the implementation light and basic where component passes 

144 run and register their own results within the :class:`ResultManager` passed in as 

145 argument. Another reason is the possibility in the near future to adopt and tap into 

146 existing and mature pass pipeline infrastructure which saves us from reinventing or 

147 rediscovering the same concepts already covered by the opensource community. 

148 """ 

149 

150 def __init__(self): 

151 self.passes: list[PassModel] = [] 

152 

153 def run(self, ir, res_mgr: ResultManager, met_mgr: MetricsManager, *args, **kwargs): 

154 """ 

155 :param arg: Argument to pass to the object. 

156 :param res_mgr: 

157 :param met_mgr: 

158 """ 

159 for p in self.passes: 

160 ir = p.run(ir, res_mgr, met_mgr, *args, **kwargs) 

161 return ir 

162 

163 def add(self, pass_obj): 

164 """Add a pass to the pass manager. 

165 

166 This can be achieved by either using :code:`pass_mgr.add(pass)`, or :code:`pass_mgr 

167 | pass`. 

168 

169 :param pass_obj: 

170 """ 

171 self.passes.append(PassModel(pass_obj)) 

172 return self 

173 

174 def __or__(self, pass_obj): 

175 return self.add(pass_obj)

178class InvokerMixin(ABC): 

179 """Useful for compilation (global design-wise) stages/phases that are meant to invoke 

180 some arbitrary formation of passes in the form of a pipeline. 

181 

182 Organisation, registering, visibility, and discovery of pipelines w.r.t the global 

183 quantum compilation workflow is hard to pin down early on in this design partly because 

184 today's needs are trivial and the traditional "hand-me-down" cascade-like compilation 

185 stages is likely to shift as QAT matures. 

186 

187 However, we'll start by specifying that an invoker builds and runs its own pipeline, 

188 validates its analyses results, and uses those results for its purpose. 

189 """ 

190 

191 @abstractmethod 

192 def build_pass_pipeline(self, *args, **kwargs) -> PassManager: 

193 pass 

194 

195 def run_pass_pipeline( 

196 self, arg, res_mgr: ResultManager, met_mgr: MetricsManager, *args, **kwargs 

197 ): 

198 return self.build_pass_pipeline(*args, **kwargs).run( 

199 arg, res_mgr, met_mgr, *args, **kwargs
